## 🎯 什么是提示词工程

### 定义

**提示词工程**（Prompt Engineering）是设计和优化输入提示词，以获得AI模型最佳输出的技术。

### 为什么重要

```
好提示词 → 准确理解 → 高质量输出
差提示词 → 理解偏差 → 低质量输出
```

**影响**：
- 📈 输出质量差异可达10倍
- ⏱️ 减少迭代次数
- 💰 降低API成本
- 🎯 提高工作效率

### 核心思想

把AI当成**新入职的聪明员工**：
- 📝 需要清晰的指令
- 🎯 需要明确的目标
- 📚 需要足够的背景
- 🌰 需要参考示例

## 📌 基础原则

### 1. 清晰明确

**原则**：具体说明你要什么

❌ **糟糕示例**：
```
写个文章
```

✅ **优秀示例**：
```
写一篇800字的科普文章，主题是"量子计算"，
目标读者是高中生，
要求：
1. 用通俗易懂的语言
2. 包含3-5个实际应用案例
3. 避免复杂的数学公式
4. 结构：引言-原理-应用-展望
```

### 2. 提供上下文

**原则**：给AI足够的背景信息

❌ **糟糕示例**：
```
优化这段代码
[代码]
```

✅ **优秀示例**：
```
这是一个Node.js后端API接口，
当前问题：响应时间超过2秒
运行环境：AWS Lambda，512MB内存
请求量：每分钟1000次

请优化这段代码，目标：
- 响应时间降到500ms以下
- 降低内存使用
- 保持代码可读性

[代码]
```

### 3. 指定格式

**原则**：明确输出格式

❌ **糟糕示例**：
```
分析这个产品
```

✅ **优秀示例**：
```
用SWOT分析法分析这个产品，
输出格式：
| 类别 | 描述 | 重要性 | 建议 |
|------|------|--------|------|
| 优势 | ... | ⭐⭐⭐ | ... |
| 劣势 | ... | ⭐⭐⭐ | ... |
| 机会 | ... | ⭐⭐⭐ | ... |
| 威胁 | ... | ⭐⭐⭐ | ... |
```

### 4. 使用示例

**原则**：给出参考样本

❌ **糟糕示例**：
```
把这个改写得正式一些
```

✅ **优秀示例**：
```
将以下内容改写成正式的商务邮件格式

示例转换：
原文：嘿，项目做完了
正式：尊敬的XX，我司已完成XX项目的开发工作

现在改写：
原文：[你的内容]
```

### 5. 分步引导

**原则**：复杂任务拆分步骤

❌ **糟糕示例**：
```
设计一个完整的电商网站
```

✅ **优秀示例**：
```
我要设计一个电商网站，请分步骤进行：

第1步：定义功能模块（只列出主要模块）
第2步：设计数据库结构（下一轮）
第3步：设计API接口（下一轮）
第4步：前端页面规划（下一轮）

现在开始第1步。
```

## 🚀 高级技巧

### 1. Few-Shot Learning（示例学习）

**原理**：给出几个示例，让AI学习模式

**模板**：
```
将产品描述转换为营销文案

示例1：
产品：高端蓝牙耳机，主动降噪
文案：沉浸纯净音质，隔绝喧嚣世界，让音乐成为你的私人空间

示例2：
产品：智能手环，24小时心率监测
文案：全天候守护健康，科学数据助力每一次运动，让健康触手可及

示例3：
产品：机械键盘，青轴，RGB背光
文案：每一次敲击都是享受，炫彩灯效点亮创作灵感，专业手感成就效率巅峰

现在转换：
产品：[你的产品]
文案：
```

### 2. Chain of Thought（思维链）

**原理**：让AI展示思考过程，提高准确性

**模板**：
```
请一步步分析以下问题，展示你的思考过程：

问题：[复杂问题]

请按以下步骤：
1. 理解问题（重新描述问题）
2. 列出相关因素
3. 分析每个因素
4. 得出结论
5. 给出建议
```

**示例**：
```
请一步步分析：如何提高网站的用户留存率？

步骤1 - 理解问题：
用户留存率是指...我们需要分析...

步骤2 - 相关因素：
- 用户体验
- 内容质量
- 加载速度
- ...

步骤3 - 因素分析：
用户体验：如果...那么...
内容质量：...

[继续分析]
```

### 3. Role Prompting（角色扮演）

**原理**：让AI扮演特定角色，获得专业视角

**示例**：
```
你是一位有20年经验的资深软件架构师，
曾在Google、Amazon等大厂工作，
擅长大规模分布式系统设计。

请从架构角度评审这个系统设计方案：
[设计方案]

重点关注：
1. 可扩展性
2. 可靠性
3. 性能
4. 维护成本
```

### 4. Self-Consistency（自我一致性）

**原理**：让AI从多个角度分析，提高准确性

**模板**：
```
请从3个不同角度分析这个问题：

问题：[问题]

角度1 - 技术可行性：
[分析]

角度2 - 商业价值：
[分析]

角度3 - 用户体验：
[分析]

综合结论：
基于以上三个角度的分析...
```

### 5. Tree of Thought（思维树）

**原理**：探索多个解决方案路径

**模板**：
```
问题：[复杂问题]

请提供3种不同的解决方案：

方案A - [方向1]：
优点：
缺点：
实施难度：

方案B - [方向2]：
优点：
缺点：
实施难度：

方案C - [方向3]：
优点：
缺点：
实施难度：

推荐：基于以上分析，推荐方案[X]，因为...
```

### 6. ReAct（推理+行动）

**原理**：结合推理和行动计划

**模板**：
```
任务：[目标]

请使用ReAct模式完成：

思考：首先我需要...
行动：第一步做...
观察：结果是...

思考：基于上一步结果...
行动：第二步做...
观察：结果是...

[继续循环]

最终结论：
```

### 案例1：产品需求文档

**任务**：为新功能写PRD

**提示词**：

任务：

1. 功能概述（100字）
2. 用户痛点（3-5个）
3. 解决方案（详细说明）
4. 核心功能列表（优先级标注）

## ⚠️ 常见错误

### 错误1：过于模糊

❌ **错误**：
```
帮我写个东西
```

✅ **正确**：
```
帮我写一份产品介绍，
产品是智能手表，
目标客户是运动爱好者，
字数300字左右
```

### 错误2：信息过载

❌ **错误**：
```
[粘贴5000字的需求文档]

帮我分析一下
```

✅ **正确**：
```
这是一份产品需求文档（5000字）

核心需求：
1. [摘要1]
2. [摘要2]
3. [摘要3]

请重点分析可行性和风险
```

### 错误3：多个任务混合

❌ **错误**：
```
写个文章，然后翻译成英文，
再总结要点，做个PPT大纲
```

✅ **正确**：
```
任务1：写文章
[具体要求]

后续任务：
- 翻译（第2轮）
- 总结（第3轮）
- PPT大纲（第4轮）
```

### 错误4：假设AI有上下文

❌ **错误**：
```
继续上面的话题
```

✅ **正确**：
```
继续讨论[具体话题]，
之前我们谈到了[具体内容]，
现在请[具体任务]
```

### 错误5：忽略验证

❌ **错误**：
```
直接使用AI生成的代码/内容
```

✅ **正确**：
```
1. 要求AI解释代码逻辑
2. 测试代码
3. 核实关键信息
4. 人工审核
```

## 📊 提示词优化流程

### 迭代优化

```
第1版：基础提示
↓
测试输出
↓
第2版：添加上下文和要求
↓
测试输出
↓
第3版：优化格式和示例
↓
最终版本
```

### 评估标准

**好提示词的特征**：
- ✅ 清晰具体
- ✅ 上下文完整
- ✅ 格式明确
- ✅ 可复现
- ✅ 输出稳定

### A/B测试

```
同一个任务，设计2个不同提示词
→ 对比输出质量
→ 选择更好的
→ 继续优化
```

## 💡 实用技巧总结

### 黄金法则

1. **具体 > 模糊**
2. **示例 > 描述**
3. **结构化 > 自由发挥**
4. **分步 > 一次完成**
5. **验证 > 盲目信任**

### 提升质量的秘诀

**3C原则**：
- **Clear**（清晰）：明确表达需求
- **Context**（上下文）：提供背景信息
- **Constraint**（约束）：设定边界条件

**SMART原则**：
- **Specific**（具体的）
- **Measurable**（可衡量的）
- **Achievable**（可实现的）
- **Relevant**（相关的）
- **Time-bound**（有时限的）

### 快速检查清单

发送提示词前检查：
- [ ] 目标明确吗？
- [ ] 上下文充足吗？
- [ ] 格式指定了吗？
- [ ] 需要示例吗？
- [ ] 任务可执行吗？
